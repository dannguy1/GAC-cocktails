# GAC Bartender Cocktail Assistant — Design Document

## 1. Problem Statement

Bartenders at Garlic & Chives need a fast, hands-friendly way to look up cocktail recipes while working behind the bar. They should be able to type a drink name — even misspelled — and instantly see the drink's image and full recipe. The environment is fast-paced, often dark, and bartenders' hands may be wet or busy.

## 2. Data Inventory

| Asset | Count | Format | Notes |
|---|---|---|---|
| Cocktail menu | 36 items | JSON (`cocktail_menu.json`) | Includes name, description, image path, price, category |
| Images | 36 files | JPG | High-quality professional photos |
| Recipes | 36 files | Markdown | Structured: Ingredients, Instructions, Serving (glass + garnish) |

**Key data observations:**

- Small dataset (36 items) — entire catalog fits in the browser with zero latency
- Recipes have consistent structure: `## Ingredients`, `## Instructions`, `## Serving`
- One duplicate entry exists (Mai Tai appears twice in JSON) — deduped at build time
- Vietnamese name/description fields exist but are currently empty
- Descriptions in JSON are abbreviated; recipe markdown files have full measurements

## 3. Core Requirements

### EARS Notation

- **WHEN** a bartender types a drink name (including misspelled input), **THE SYSTEM SHALL** display matching cocktails ranked by similarity within 100ms.
- **WHEN** a bartender selects a cocktail, **THE SYSTEM SHALL** display the drink image, full ingredient list, step-by-step instructions, glass type, and garnish in a split-screen layout.
- **WHILE** the app is running in a bar environment, **THE SYSTEM SHALL** use a dark theme with high-contrast text optimized for low-light conditions.
- **THE SYSTEM SHALL** work fully offline after the first load (all 36 cocktails cached locally via service worker).
- **IF** no matching cocktail is found, **THE SYSTEM SHALL** display a "No matches found" message.
- **THE SYSTEM SHALL** be available as both a web PWA and a React Native/Expo mobile app sharing the same design system and data.

## 4. Implemented Features

### 4.1 "Instant Pour" Fuzzy Search

Client-side fuzzy matching via **Fuse.js** (v7). The bartender never needs to spell perfectly — results appear as-you-type with 100ms debounce and zero network delay.

- Typing "mrgrita" or "old fashin" or "moheeto" all resolve instantly
- Search bar is always visible in the header, auto-focused on load
- Keyboard navigation: Arrow Up/Down to highlight, Enter to select, Escape to close
- Search keys: `name` (weight 1.0), `aliases` (weight 0.8), `ingredients.raw` (weight 0.3)
- Threshold: 0.5 (allows significant typos while avoiding nonsense matches)
- XSS protection: all data interpolated into HTML is escaped via a dedicated `esc()` helper

**Alias matching:** Common bar slang and abbreviations mapped at build time:

- "LIT" or "LIIT" → Long Island Tea
- "OF" → Old Fashioned
- "marg" → Margarita
- "mule" → Moscow Mule, Irish Mule

### 4.2 Dark "Bar Mode" UI

Entire interface designed for dark environments:

- **Near-black background** (`#0f0a06`) with warm brand gold accents (`#c6893f`)
- **Large, legible typography** — 18px body text, drink names use `clamp(1.5rem, 3vw, 2.25rem)`
- **High-contrast ingredient list** — light text on dark surface with checklist indicators
- **No bright white surfaces** — all panels, dropdowns, and modals use dark surfaces

### 4.3 Interactive Ingredient Checklist

Ingredients displayed as a tappable checklist:

```
 ☑ 2 oz tequila (blanco)
 ☐ 1 oz fresh lime juice
 ☐ 1 oz Cointreau or triple sec
 ☐ ½ oz agave syrup
 ☐ Salt for rimming
```

- Tap to check/uncheck; checked items dim with strikethrough
- State persisted in `localStorage` (web) or `AsyncStorage` (mobile) per cocktail ID
- ARIA `role="checkbox"` with `aria-checked` for accessibility
- 48px minimum touch target per row

### 4.4 Split-Screen Recipe Card

When a cocktail is selected, it displays in a two-panel split layout (landscape):

- **Left panel:** Drink image (4:3 aspect ratio, `object-fit: cover`), category badge overlay, name, price, glass, garnish, description
- **Right panel:** Ingredient checklist and step-by-step instructions

In portrait / small screens (`max-width: 768px`), panels stack vertically.

### 4.5 "Recent Drinks" Quick Access

A persistent strip at the bottom of the screen showing the last 5 drinks (web) or 8 drinks (mobile):

- One tap to re-open a recipe
- Stored in `localStorage` (web) / `AsyncStorage` (mobile) — persists across sessions
- 72px thumbnails in 88px-wide cards with drink name below
- Horizontal scroll when more items than visible space

### 4.6 Progressive Web App (PWA) — Works Offline

- **Service Worker** generated by `vite-plugin-pwa` (Workbox) with `autoUpdate` registration
- Precaches all images, scripts, styles, and HTML (~2.3 MB total, 45 entries)
- Installs to home screen as standalone app (no browser chrome)
- Full functionality with zero network dependency after initial load
- `display: standalone`, `orientation: any`

### 4.7 React Native / Expo Mobile App

A companion mobile app built with React Native 0.81 + Expo ~54, sharing the same cocktail data and design system:

- Split-screen layout in landscape, stacked in portrait (mirrors web behavior)
- Images served from the web app's Vite server at a configurable host (default `http://192.168.10.3:8508`)
- `react-native-safe-area-context` for safe area handling
- `@react-native-async-storage/async-storage` for persistence
- Fuse.js for identical fuzzy search behavior
- APK build script (`build_apk.sh`) following the GAC-Concierge pattern

### 5.2 Data Pipeline

```
cocktail_menu.json ─┐
                     ├──▶ build-data.js ──▶ src/data/cocktails.js (ES module)
recipes/*.md ────────┘                           │
                                                 ├──▶ Vite bundle (web)
images/*.jpg ──▶ public/images/                  │
                                                 └──▶ build-mobile-data.js ──▶ mobile/data/cocktails.json
```

**Build scripts:**

| Script | Command | Output |
|---|---|---|
| `npm run build:data` | `node --env-file=.env scripts/build-data.js` | `src/data/cocktails.js` |
| `npm run build:mobile-data` | `node scripts/build-mobile-data.js` | `mobile/data/cocktails.json` |
| `npm run build` | `build:data && vite build` | `dist/` (production bundle) |
| `npm run dev` | `vite` | Dev server with HMR |
| `npm run start` | `vite preview --port 8508` | Production preview server |

**Build-time preprocessing (`build-data.js`):**

1. Read `cocktail_menu.json` from `COCKTAIL_DATA_DIR` — extract name, price, description, image path
2. Deduplicate by `item_name` (Mai Tai appears twice — keep first)
3. Read each `recipes/*.md` — parse into `{ ingredients[{ raw }], instructions, glass, garnish }`
4. Merge JSON metadata with parsed recipes by name
5. Attach aliases from hardcoded alias map (~15 mappings)
6. Write `src/data/cocktails.js` as an ES module exporting the array
7. Copy images to `public/images/`

**Mobile data pipeline (`build-mobile-data.js`):**

1. Read `src/data/cocktails.js` ES module source
2. Extract the `export default` array via regex match
3. Parse as JSON, optionally rewrite image paths with `--host` flag
4. Write `mobile/data/cocktails.json`

### 5.3 Search Algorithm Design

```
User input: "moheeto"
          │
          ▼
    ┌─────────────┐
    │  Fuse.js     │  Keys: [name, aliases, ingredients.raw]
    │  threshold:  │  Weight: name=1.0, aliases=0.8, ingredients=0.3
    │  0.5         │
    └──────┬──────┘
           │
           ▼
    Ranked results (up to 5 on web, 8 on mobile):
    1. Mojito        (best match)
    2. Moscow Mule
    3. Mai Tai
           │
           ▼
    Dropdown with thumbnail + name;
    tap/click or Enter to select
```

**Search configuration (both platforms):**

- Search keys: `name` (weight 1.0), `aliases` (weight 0.8), `ingredients.raw` (weight 0.3)
- Threshold: 0.5
- Max results: 5 (web) / 8 (mobile)
- Minimum match character length: 1
- Debounce: 100ms (web) / 250ms (mobile)

### 5.5 Component Architecture

#### Web Components (Shadow DOM)

| Component | Element | Responsibilities |
|---|---|---|
| `gac-search-bar` | `<gac-search-bar>` | Text input, Fuse.js search, debounced results dropdown, keyboard nav, XSS escaping |
| `gac-recipe-card` | `<gac-recipe-card>` | Split-screen display, image + info left, recipe right, empty state |

**Event flow:** `gac-search-bar` dispatches `cocktail-selected` → `main.js` sets `recipeCard.cocktail` and calls `recentStrip.addRecent()`.

#### Mobile Components (React)

| Component | File | Responsibilities |
|---|---|---|
| `App` | `App.js` | Root layout, state management, `useRef` for stale closure avoidance |

| `IngredientChecklist` | `components/IngredientChecklist.js` | Tap-to-check, AsyncStorage persistence per cocktail |

**Services:**

| Module | Purpose |
|---|---|
| `services/search.js` | Fuse.js wrapper — `searchCocktails()`, `getCocktailById()`, `getAllCocktails()` |
| `services/storage.js` | AsyncStorage CRUD for recent drinks, checklists, web host config; `resolveImageUrl()` for image paths |

### 5.6 Security

- **XSS prevention:** All web components that use `innerHTML` with data pass values through an `esc()` helper (`document.createElement('div').textContent = str; return div.innerHTML`)
- **No server-side code** — no injection surface for SQL, command, or SSRF attacks
- **Content Security Policy** inherits from static hosting provider

## 6. User Flows

### 6.1 Primary Flow — Search by Name

1. App loads → search input auto-focused
2. Bartender types partial/misspelled name → results dropdown appears after 100ms debounce
3. Results show thumbnail + name; Arrow keys or tap to navigate
4. Select a match → split-screen recipe card appears with image left, recipe right
5. Bartender taps ingredients to check them off as they pour
6. Drink appears in the "Recent" strip at the bottom

### 6.2 Recent Drink Flow

1. Bartender spots a previously made drink in the recent strip
2. One tap → recipe card loads instantly (no search needed)
3. Checklist state is restored from storage for that cocktail

### Card Pattern

- **Split layout** in landscape: image + info left, recipe right (differs from original single-column design)
- Drink name: `font-family: var(--font-serif)`, `color: var(--color-brand-gold-light)`
- Price: `font-weight: 700`, `color: var(--color-brand-gold-light)`

### Motion

- Recipe card fades in on drink switch (200ms ease)

## 8. Accessibility & Durability

- **Touch targets:** Minimum 48x48px for all interactive elements (enforced via `--touch-target-min` CSS variable and StyleSheet)
- **High contrast:** Light text on dark backgrounds throughout
- **Splash-proof design:** No hover-only interactions; everything works with tap
- **Keyboard accessible:** Arrow key navigation in search results, Enter to select, Escape to close
- **ARIA attributes:** `role="listbox"` on results, `role="checkbox"` + `aria-checked` on ingredient rows, `aria-label` on search input
- **Resilient to wet taps:** No swipe-to-delete or gestures that misfire with wet fingers

## 9. Decisions Made

| # | Question | Decision | Rationale |
|---|---|---|---|
| 1 | Web framework | Vanilla JS + Web Components | 36 items; no framework overhead needed. Shadow DOM for encapsulation. |
| 2 | Mobile framework | React Native + Expo | Cross-platform (Android/iOS/web). Follows GAC-Concierge pattern. |
| 3 | Recipe rendering | Build-time (markdown → JS object) | Faster startup, simpler runtime, no markdown parser in bundle. |
| 4 | Voice search | Deferred (future) | MVP focuses on typed search; voice adds complexity. |
| 5 | Layout | Split-screen (landscape) | Tablet-optimized: image + info left, recipe right. Stacks in portrait. |
| 6 | Vietnamese fields | Future phase | Fields exist but are empty in source data. |
| 7 | Image optimization | Copy JPGs as-is | 36 images are already sized; WebP conversion can be added later. |
| 8 | Deployment (web) | Static via Vite preview server | `vite preview --port 8508` serves the production build. |
| 9 | Alias dictionary | Hardcoded in build script | Small set (~15 aliases); easy to maintain. |
| 10 | Mobile images | Served from web server | Mobile app loads images from the Vite server via `resolveImageUrl()`. Avoids bundling images in the APK. |
| 11 | Visual identity | GAC ecosystem dark variant | Reuse brand-gold/green palette from GAC-Concierge/display. |
| 12 | Data path | `COCKTAIL_DATA_DIR` env var in `.env` | No hardcoded paths; `.env.example` for setup. |
| 13 | Mobile fonts | Platform defaults (`undefined`) | Avoids cross-platform font loading issues; uses OS serif/sans. |
| 14 | XSS prevention | `esc()` helper in all web components | All `innerHTML` interpolations pass through `textContent`-based escaping. |
| 15 | Stale closure | `useRef` for mutable state in callbacks | `recentIdsRef.current` avoids stale captures in `handleSelect`. |
| 16 | Search threshold | 0.5 (was 0.4 in original design) | Tuned during implementation for better balance of recall vs precision. |
